ChatGPT said:

Attaching a bump sensor (typically represented by a contact sensor in Gazebo) to a wall in a ROS 2 + Gazebo simulation setup involves several steps. Here’s a clear and concise guide to achieve this:

1. Create or Modify a Wall Model

You can either use an existing wall model or create a custom one using SDF or URDF.

Example (SDF-based Wall):

xml
<model name="wall_with_bumper">
  <static>true</static>
  <link name="wall_link">
    <collision name="collision">
      <geometry>
        <box>
          <size>2 0.1 1</size> <!-- width, depth, height -->
        </box>
      </geometry>
    </collision>
    <visual name="visual">
      <geometry>
        <box>
          <size>2 0.1 1</size>
        </box>
      </geometry>
    </visual>

    <!-- Contact Sensor -->
    <sensor name="bump_sensor" type="contact">
      <always_on>true</always_on>
      <update_rate>50</update_rate>
      <contact>
        <collision>collision</collision>
      </contact>
      <plugin name="contact_plugin" filename="libgazebo_ros_contact.so">
        <ros>
          <namespace>/wall</namespace>
          <remapping>~/contacts:=bumper</remapping>
        </ros>
      </plugin>
    </sensor>
  </link>
</model>

2. Use the Gazebo ROS Contact Plugin

Ensure you are using the gazebo_ros_contact plugin (part of gazebo_ros_pkgs). It publishes contact events to a ROS 2 topic.

The above example sets it to publish to /wall/bumper.

3. Add to World or Launch with ROS 2

If adding directly to a world file:

xml
<include>
  <uri>model://your_wall_model</uri>
</include>

If launching with ROS 2:

  • Make sure your model is in the correct path.

  • Launch using gazebo_ros with the model inserted, or spawn it using a node like gazebo_ros_factory.

4. Subscribe to the Bump Sensor Topic in ROS 2

The sensor publishes gazebo_msgs/msg/ContactsState:

bash
ros2 topic echo /wall/bumper

This will show contact data when the wall is touched.

5. Tips

  • Make sure the wall and sensor are not in collision at spawn.

  • Use rqt_graph or ros2 topic list to confirm topics are live.

  • If you use URDF + xacro, you can include similar <gazebo> tags within your link/joint definitions.

Great question! You're referring to a Gazebo (Ignition/LibGazebo) Touch Plugin, which is part of the new Gazebo (formerly Ignition Gazebo) plugin system, not the legacy Gazebo Classic one.

This TouchPlugin is used in Gazebo Sim (Fortress, Harmonic, etc.), and it's not the same as the libgazebo_ros_contact.so plugin used with Gazebo Classic + ROS 2. Here's how to understand and use that plugin:

🔧 What TouchPlugin Does

This plugin detects contacts and publishes them to Gazebo topics. It doesn’t directly publish to ROS 2 topics unless bridged (e.g., with ros_gz_bridge).

Here's what each tag means:

xml
<plugin filename="gz-sim-touchplugin-system"
        name="gz::sim::systems::TouchPlugin">
  <target>mmtr</target>             <!-- Link or collision element to monitor -->
  <namespace>wall</namespace>       <!-- ROS 2 namespace or Gazebo topic prefix -->
  <time>0.001</time>                <!-- Update interval in seconds -->
  <enabled>true</enabled>           <!-- Whether it's active at start -->
</plugin>

✅ To Use It with ROS 2 + Gazebo Sim (Ignition)

  1. Ensure You Use gazebo_ros_pkgs and ros_gz

    • Install ros_gz_bridge

    • Install ros_gz_sim plugins

  2. Use this plugin in your SDF model
    Place it inside a <link> element of the wall model that includes a collision.

  3. Bridge the Gazebo Topic to ROS 2
    The TouchPlugin will publish to a Gazebo topic like:

    bash
    /wall/touch

    You’ll need to bridge that topic to ROS 2. For example:

    bash
    ros2 run ros_gz_bridge parameter_bridge /wall/touch@gz.msgs.ContactsState@ros_gz_interfaces/msg/ContactsState

  4. Subscribe in ROS 2

    Once bridged, you can:

    bash
    ros2 topic echo /wall/touch

⚠️ Notes

  • The <target> should be the link or collision name you want to monitor.

  • Ensure that the link/collision is properly defined and capable of contact detection.

  • If nothing shows up, double-check your collisions and that something is actually making contact.

🔴 Why It Fails

  • The TouchPlugin is a model system plugin.

  • It must be attached to a model in the SDF, not directly to <world>.

  • It cannot monitor other models from the world file — Gazebo plugins can't "reach into" other models like that.

✅ How to Fix It

You need to move the plugin into a model SDF, and spawn that model near or around the wall.

Example: Create a Bump Sensor Model

Make a new model (e.g., bump_wall.sdf):

xml
<model name="wall_with_bump_sensor">
  <pose>0 0 0 0 0 0</pose>
  <link name="sensor_link">
    <pose>0 0 0.5 0 0 0</pose>
    <collision name="collision">
      <geometry>
        <box><size>0.5 0.1 1.0</size></box>
      </geometry>
      <surface>
        <contact><collide_without_contact>true</collide_without_contact></contact>
      </surface>
    </collision>
    <visual name="visual">
      <geometry>
        <box><size>0.5 0.1 1.0</size></box>
      </geometry>
    </visual>
  </link>

  <plugin name="gz::sim::systems::TouchPlugin"
          filename="gz-sim-touchplugin-system">
    <target>sensor_link</target>
    <namespace>wall</namespace>
    <time>0.001</time>
    <enabled>true</enabled>
  </plugin>
</model>

Then include this model in your world, or spawn it with ros_gz_sim’s spawn service.

🔄 Alternatives

If you want to detect collisions on your robot, move the TouchPlugin into the robot model's SDF or xacro directly, e.g.:

xml
<gazebo>
  <plugin name="gz::sim::systems::TouchPlugin"
          filename="gz-sim-touchplugin-system">
    <target>my_robot_link</target>
    <namespace>my_robot</namespace>
    <time>0.001</time>
    <enabled>true</enabled>
  </plugin>
</gazebo>

This must be within the robot’s <model> and target a valid link or collision element.

xml

✅ For Gazebo Sim (Ignition) + ROS 2

If you're using Gazebo Sim (formerly Ignition) (i.e. ros_gz_sim), the <sensor> won't publish by itself. You must:

  1. Remove <sensor> (not supported the same way).

  2. Use a system plugin instead:

Add the TouchPlugin inside the <model>:

xml
<plugin filename="gz-sim-touchplugin-system"
        name="gz::sim::systems::TouchPlugin">
  <target>wall::box</target>
  <namespace>wall</namespace>
  <time>0.001</time>
  <enabled>true</enabled>
</plugin>

  • target must include the full scoped name: model_name::link_name

  • You can also just use box if the plugin is already inside the wall model — it’s context-aware.

Then bridge it with ros_gz_bridge:

bash
ros2 run ros_gz_bridge parameter_bridge /wall/touch@gz.msgs.ContactsState@ros_gz_interfaces/msg/ContactsState

Now you'll be able to:

bash
ros2 topic echo /wall/touch

ChatGPT said:

Exactly — the target in the TouchPlugin is not the robot. It's the part of the model where the plugin is attached and where contact will be detected.

🔍 Think of it this way:

  • You attach the TouchPlugin to your wall model.

  • The plugin can only monitor collisions on the same model — in this case, the wall.

  • So the target should be a link or collision inside the wall — not the robot.

✅ Example

xml
<model name="wall">
  ...
  <link name="box">
    <collision name="wall_collision">
      ...
    </collision>
    ...
  </link>

  <plugin name="gz::sim::systems::TouchPlugin"
          filename="gz-sim-touchplugin-system">
    <target>box</target>              <!-- ✅ A link or collision inside 'wall' -->
    <namespace>wall</namespace>
    <time>0.001</time>
    <enabled>true</enabled>
  </plugin>
</model>

What Happens

  • If your robot hits the wall, and the wall’s box link experiences a collision, the TouchPlugin will detect it.

  • The plugin does not care what hit the wall — just that there was a contact involving the target.

🚫 Invalid Example (why it fails)

If you set:

xml
<target>robot::link</target>

in the wall model's plugin, it fails because:

  • robot::link is not part of the wall model

  • The plugin is scoped only to the model it’s attached to

This results in:

[Err] Touch plugin should be attached to a model entity. Failed to initialize.
