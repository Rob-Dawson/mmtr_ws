Is there a way of using the ROS2 occupancy grid topic without using a Lidar? I want to build an occupancy grid but it will use a crash sensor for obstacles. Crash black cell, no crash, white cell
ChatGPT said:

Yes, you can definitely use ROS2 OccupancyGrid without a LiDAR in fact, ROS2 is sensor-agnostic. You just need to publish the right data to build and update the occupancy grid. If you're using a crash sensor (e.g., bumper or contact sensor), you can create a very simple map using your own logic.

How to do it:

Heres a general approach to building an occupancy grid with crash sensors:

1. Understand the nav_msgs/OccupancyGrid format

Youll need to publish a message of type nav_msgs/msg/OccupancyGrid, which includes:

  • header timestamp and frame ID.

  • info metadata like resolution, width, height, and origin.

  • data a flattened array of int8 values:

    • -1: unknown

    • 0: free

    • 100: occupied

2. Create your own occupancy grid node

You can write a custom ROS2 node (e.g., in Python or C++) that:

  • Maintains a 2D grid in memory.

  • Updates grid cells based on bumper/crash sensor feedback.

  • Publishes the updated grid at a fixed rate.

3. How to Update the Grid with a Crash Sensor

Assuming you have:

  • A way to determine the robots position (e.g., from odometry or AMCL).

  • A binary crash signal.

Logic:

python
if crash_detected:
    mark the cell in front of the robot as OCCUPIED (100)
else:
    mark it as FREE (0)

You need to:

  • Project the robots pose + heading to the grid.

  • Mark the cell in that direction accordingly.

  • Over time, you can build up the map.

4. ROS2 Code Skeleton (Python, rclpy)

python
import rclpy
from rclpy.node import Node
from nav_msgs.msg import OccupancyGrid
from geometry_msgs.msg import Pose, Point, Quaternion
import numpy as np

class BumperGridMapper(Node):
    def __init__(self):
        super().__init__('bumper_grid_mapper')

        # Grid config
        self.resolution = 0.1  # meters per cell
        self.width = 100
        self.height = 100
        self.origin = [-5.0, -5.0]  # meters

        self.grid = np.full((self.height, self.width), -1, dtype=np.int8)  # unknown

        # Publisher
        self.pub = self.create_publisher(OccupancyGrid, 'map', 10)

        # Timer
        self.timer = self.create_timer(1.0, self.publish_map)

        # Replace with your actual subscription
        # self.create_subscription(... crash sensor ...)
        # self.create_subscription(... odometry or tf ...)

    def update_cell(self, x_m, y_m, occupied):
        x_idx = int((x_m - self.origin[0]) / self.resolution)
        y_idx = int((y_m - self.origin[1]) / self.resolution)

        if 0 <= x_idx < self.width and 0 <= y_idx < self.height:
            self.grid[y_idx, x_idx] = 100 if occupied else 0

    def publish_map(self):
        msg = OccupancyGrid()
        msg.header.frame_id = 'map'
        msg.info.resolution = self.resolution
        msg.info.width = self.width
        msg.info.height = self.height
        msg.info.origin.position.x = self.origin[0]
        msg.info.origin.position.y = self.origin[1]
        msg.info.origin.orientation.w = 1.0
        msg.data = self.grid.flatten().tolist()
        self.pub.publish(msg)

def main(args=None):
    rclpy.init(args=args)
    node = BumperGridMapper()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()

If your rover is exploring miles of forestry, using a crash sensor alone to build an occupancy grid becomes more challenging but not impossible. The main issues to address are:

Challenges in Large-Scale Mapping with Crash Sensors

1. Map Size & Memory Limits

ROS2s nav_msgs/OccupancyGrid is static-sized. For miles of forest, youd need a grid that's:

  • Too big to fit in memory if you use high resolution (e.g., 0.1 m/cell),

  • Unusable for practical visualization in RViz or path planning tools.

2. Lack of Range

Crash sensors only detect obstacles on contact. Unlike LiDAR or sonar, they dont provide information about nearby space unless you hit something. This makes proactive path planning or detailed mapping very difficult.

3. Localization

To build a coherent map over miles, your rover must have reliable localization, e.g., via:

  • GPS (for rough global location),

  • IMU + wheel odometry (for dead reckoning),

  • SLAM (though this usually needs LiDAR or camera),

  • Visual odometry (using a monocular or stereo camera).

Solutions for Forest-Scale Mapping with Crash Sensors

1. Use a Rolling Occupancy Grid or Tiled Map

Instead of mapping the entire forest in one huge grid:

  • Use a sliding local map around the robot (e.g., 10m x 10m window),

  • Or implement a tiled/map chunk system like in Minecraft or OpenStreetMap:

    • Store map tiles on disk or in a database,

    • Load and unload tiles as the robot moves.

Libraries like grid_map, map_server, or even custom code with sqlite/rosbag can help.

2. Integrate GPS + IMU + Odometry

Use GPS for global coordinates:

  • Record each crash event as a geospatial coordinate,

  • Build a sparse occupancy map in lat/lon (or convert to UTM or ENU frame).

Even if you can't publish an OccupancyGrid, you could:

  • Build a point-based obstacle map (e.g., a geojson or KD-tree),

  • Or render dynamic maps using tools like RViz, PointCloud2, or custom visuals.

3. Add Passive Sensors for More Coverage

Even cheap sensors like:

  • Ultrasonic rangefinders,

  • Monocular cameras (with depth estimation),

  • Or wheel encoders + IMU for motion estimation

can drastically improve mapping. This isnt a requirement, but it will help fill in the gaps between collisions.

4. Use Semantic Maps or Trail Mapping

If you're mapping trails in a forest:

  • Dont map every tree just the navigable path.

  • You could record:

    • GPS trails,

    • Crash events as blocked regions,

    • Open zones as traversable.

Then use tools like:

  • rosbag2 for logging,

  • GeoJSON, OccupancyGrid, or even a GIS-based backend for visualization.

While a crash sensor alone wont give you full 3D or long-range mapping, it can be used to:

  • Build local occupancy grids,

  • Mark known obstacles,

  • Provide sparse maps of unstructured environments.

If you're working on a long-range, low-cost autonomous rover, this approach is very feasible with smart engineering and adaptive mapping logic.

ChatGPT said:

Great question and you're absolutely right to spot this.

Short Answer:

You should absolutely put the map initialization in the constructor not in publish_map() because:

  • The grid only needs to be created once (when the node starts),

  • publish_map() is called repeatedly (e.g., via a timer), so reinitializing it there would reset the map every time, erasing any updates.

In Detail:

Lets compare the two designs:

Bad (map initialized in publish_map()):

python
def publish_map(self):
    self.grid = np.full((self.height, self.width), -1, dtype=np.int8)  # BAD
    # ...
    msg.data = self.grid.flatten().tolist()
    self.pub.publish(msg)

Problem:

  • self.grid gets reset to -1 (unknown) every time publish_map() is called.

  • All crash-based or free-space updates would be lost every second.

Good (map initialized in __init__() constructor):

python
def __init__(self):
    super().__init__('bumper_grid_mapper')
    
    # Grid initialization (ONE-TIME)
    self.grid = np.full((self.height, self.width), -1, dtype=np.int8)

    # Other setup...
    self.pub = self.create_publisher(OccupancyGrid, 'map', 10)
    self.timer = self.create_timer(1.0, self.publish_map)

Result:

  • The occupancy grid is persistent.

  • You can update it (e.g., using bumper/crash events), and publish_map() just reflects the current state without erasing it.

Best Practice Pattern

python
# In __init__()
self.grid = np.full((self.height, self.width), -1, dtype=np.int8)

# In update_cell(x, y, state)
self.grid[y, x] = 0 or 100

# In publish_map()
msg.data = self.grid.flatten().tolist()
self.pub.publish(msg)

This separation ensures:

  • One-time setup stays in __init__(),

  • Continuous updates happen via event/callback logic,

  • Publishing is read-only with respect to the map state.

Final Takeaway

You're 100% correct initializing the map in publish_map() is a logic bug unless you're doing rolling windows or time-based decay (which would need different logic). For persistent occupancy mapping, constructor initialization is the way to go.
